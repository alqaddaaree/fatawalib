# Fatāwā Archive

A clean, fast static website for publishing translated Islamic fatwas, built with [Astro](https://astro.build), Tailwind CSS, and Fuse.js for client-side search.

## Features

- 📚 **Content Collections** — Type-safe frontmatter with Zod schema validation
- 🔍 **Full-text search** — Client-side Fuse.js search over a pre-built JSON index
- **Scholar pages** — Dedicated pages for each scholar with bio and fatwa listings
- 🗂️ **Category pages** — Browse fatwas by topic
- 🎨 **Clean Tailwind UI** — Teal/cream/gold palette with Libre Baskerville + Source Sans 3
- ⚡ **Static output** — Fully pre-rendered, deployable anywhere
- 🚀 **Vercel-ready** — `vercel.json` included

---

## Getting Started

```bash
npm install
npm run dev
```

Open [http://localhost:4321](http://localhost:4321).

## Adding Fatwas

Create a new Markdown file in `src/content/fatwas/`:

```
src/content/fatwas/006.md
```

With this frontmatter:

```yaml
---
id: "006"
scholar: ibn-baz              # ibn-baz | ibn-uthaymin | al-albani | muqbil | yahya
categories:
  - Prayer
  - Purification
source_url: https://binbaz.org.sa/fatwas/5678
translator: Abu Abdillah
reviewer: Abu Khadeejah       # optional
date_issued: "1415-03-10"    # Hijri or ISO date string
date_added: 2024-06-01       # ISO date — used for sorting
---

## Question

Your question text here.

## Answer

The scholar's answer here.
```

The `id` should match the filename (without `.md`) and be unique.

The search index is generated automatically during build. To regenerate manually:

```bash
npm run build:index
```

This reads all Markdown files and writes `public/search-index.json` used by the client-side Fuse.js search.

### Key mental model

There are two kinds of data in this project:

1. **Content** — the actual fatwas — lives entirely in `src/content/fatwas/*.md`. Each file is one fatwa. You never touch TypeScript or HTML to add content.
2. **Structure** — the pages, components, and logic — lives in `src/`. You only touch these files when you want to change the _site itself_, not when you add fatwas.

## 2. Content Collection Configuration

**File:** `src/content/config.ts`

```ts
import { defineCollection, z } from 'astro:content';

const scholars = ['ibn-baz', 'ibn-uthaymin', 'al-albani', 'muqbil', 'yahya'] as const;

const fatwasCollection = defineCollection({
  type: 'content',
  schema: z.object({
    id:          z.string(),
    scholar:     z.enum(scholars),
    categories:  z.array(z.string()),
    source_url:  z.string().url(),
    translator:  z.string(),
    reviewer:    z.string().optional(),
    date_issued: z.string(),
    date_added:  z.coerce.date(),
  }),
});

export const collections = {
  fatwas: fatwasCollection,
};
```

This file does two things: it tells Astro that the `fatwas/` folder is a "Content Collection", and it defines a **schema** — a strict set of rules about what fields every fatwa file must have. If you add a file that violates these rules (e.g. misspell a scholar's ID), the build will fail with a clear error message.

### Field-by-field breakdown

| Field | Type | Required? | What it does |
|---|---|---|---|
| `id` | string | **Yes** | A unique identifier for the fatwa, e.g. `"006"`. Used in the URL: `/fatwa/006`. Must be unique across all files. |
| `scholar` | one of 5 fixed values | **Yes** | Must be exactly one of: `ibn-baz`, `ibn-uthaymin`, `al-albani`, `muqbil`, `yahya`. Any other value causes a build error. |
| `categories` | array of strings | **Yes** | A YAML list of category names, e.g. `["Prayer", "Purification"]`. Free-form — invent new categories freely. |
| `source_url` | string (valid URL) | **Yes** | Must be a full, valid URL (with `https://`). Astro validates the format at build time. |
| `translator` | string | **Yes** | The name of the English translator. |
| `reviewer` | string | **No** | Optional reviewer name. If omitted entirely, it simply won't appear on the fatwa page. |
| `date_issued` | string | **Yes** | The date the original fatwa was issued. Stored as a plain string, so you can use Hijri dates like `"1415-03-10"` or anything descriptive. Not parsed as a date. |
| `date_added` | date | **Yes** | The date this translation was added to the site. **Must be a valid ISO date** like `2024-06-01` because Astro parses it as a JavaScript `Date` object (using `z.coerce.date()`). This is what controls the sort order on the homepage. |

### What `z.coerce.date()` means

`date_added` uses Zod's `coerce.date()`, which means: "take whatever string is in the YAML and convert it to a JavaScript Date object." This is why you write `date_added: 2024-06-01` (no quotes) in the file but the code can call `.getTime()` on it to sort by date. If you write an invalid date like `2024-13-99`, the build will error.

### The `scholars` enum

The list `['ibn-baz', 'ibn-uthaymin', 'al-albani', 'muqbil', 'yahya']` is a TypeScript `const` array that becomes a Zod enum. This means only these exact lowercase-hyphenated strings are valid. If you type `Ibn-Baz` or `ibn_baz` it will fail. See [Section 12](#12-common-maintenance-scenarios) for how to add a new scholar.

## 3. A Sample Fatwa File Dissected

Let's use `src/content/fatwas/001.md` as our example:

```markdown
---
id: "001"
scholar: ibn-baz
categories:
  - Prayer
  - Purification
source_url: https://binbaz.org.sa/fatwas/1234
translator: Abu Abdillah
reviewer: Abu Khadeejah
date_issued: "1415-03-10"
date_added: 2024-01-15
---

## Question

Is it permissible to pray while wearing shoes, and what is the condition for that?

## Answer

It is permissible to pray while wearing shoes if they are free from impurity...

And Allāh knows best.
```

### The three-dash boundary

The `---` lines are the **frontmatter delimiters**. Everything between the first `---` and the second `---` is YAML (structured data). Everything **after** the second `---` is the Markdown body.

```
---          ← start of frontmatter
id: "001"    ← YAML data
...
---          ← end of frontmatter
             ← blank line (good practice)
## Question  ← start of Markdown body
```

### Body structure conventions

The body uses standard Markdown with two expected headings:

- `## Question` — the question as it was posed to the scholar
- `## Answer` — the scholar's response

These headings are not enforced by code — they're a convention. The `##` syntax produces an `<h2>` HTML element, which the fatwa page styles with a bottom border in teal color. You could add other `##` sections if needed (e.g. `## Notes`, `## Source Context`), and they'd render the same way.

**Markdown formatting you can use in the body:**

| Syntax | Result |
|---|---|
| `*text*` or `_text_` | *italic* |
| `**text**` | **bold** |
| `> blockquote` | an indented block quote |
| `- item` or `* item` | bullet list |
| blank line between paragraphs | new paragraph |
| `[link text](https://url)` | hyperlink |

The Prophet symbol `ﷺ` is a Unicode character — you can copy-paste it directly into the Markdown.

## 4. How to Add a New Fatwa

### Step 1 — Create the file

Create a new `.md` file inside `src/content/fatwas/`. The filename should match the `id` you plan to use:

```
src/content/fatwas/006.md
```

You can name it whatever you want (the filename itself doesn't matter to the router), but matching filename to `id` keeps things organized.

### Step 2 — Fill in the frontmatter

Copy and paste this template:

```yaml
---
id: "006"
scholar: ibn-baz
categories:
  - Prayer
source_url: https://binbaz.org.sa/fatwas/XXXX
translator: Your Translator Name
date_issued: "1420-05-10"
date_added: 2024-07-01
---
```

Required fields you must always fill:
- `id` — unique string, wrap in quotes to be safe
- `scholar` — must be exactly one of the five allowed values
- `categories` — at least one category as a YAML list item (start the line with two spaces then `- `)
- `source_url` — full URL including `https://`
- `translator` — any string
- `date_issued` — any string (Hijri or Gregorian)
- `date_added` — ISO date `YYYY-MM-DD` (no quotes)

Optional:
- `reviewer` — add only if there is one

### Step 3 — Write the body

After the closing `---`, leave a blank line, then write the content:

```markdown
## Question

Write the question here. Keep it as close to the original as possible.

## Answer

Write the scholar's answer here, in as many paragraphs as needed.

You can use *italics* for Arabic terms and **bold** for emphasis.

And Allāh knows best.
```

### Step 4 — Dev mode: immediate appearance

In development mode (`npm run dev`), **new fatwa files appear immediately** after you save. Astro hot-reloads content. You do not need to restart the server.

### Step 5 — Regenerate the search index

**This is the one step people forget.** The search index (`public/search-index.json`) is a static file generated by a script. In dev mode, the search feature reads from whatever is currently in that file — it does not auto-update when you add content.

To update the search index in development:

```bash
npm run build:index
```

This runs `scripts/build-search-index.js` and rewrites `public/search-index.json`. After that, refresh the search page and the new fatwa is searchable.

When you run `npm run build` (for production), the script runs automatically first — so the production build always has a fresh index.

### Step 6 — Verify

Open `http://localhost:4321` and the new fatwa card should appear on the homepage. Open `http://localhost:4321/fatwa/006` (replace with your id) directly to see the full page.

### Summary checklist

- [ ] Create `src/content/fatwas/006.md`
- [ ] Fill all required frontmatter fields
- [ ] Write `## Question` and `## Answer` sections
- [ ] Confirm `date_added` is ISO format `YYYY-MM-DD`
- [ ] Confirm `scholar` is one of the 5 exact allowed values
- [ ] Run `npm run build:index` (dev) or just `npm run build` (production)

## 5. How to Edit an Existing Fatwa

### Editing the text

Open the relevant `.md` file in `src/content/fatwas/` and edit the body text. In dev mode (`npm run dev`), changes appear immediately in the browser — you don't even need to refresh manually on most systems (Astro uses HMR, Hot Module Replacement).

### Editing frontmatter fields

You can change any frontmatter field. A few things to know:

- **Changing `categories`**: Adding or removing a category string is fine. The category pages are generated dynamically from whatever categories exist in the frontmatter — new categories get their own `/category/` page automatically at the next build.
- **Changing `scholar`**: Allowed, but only to one of the 5 valid values. This will move the fatwa to a different scholar page.
- **Changing `id`**: Possible, but **will break any existing links** to the old URL (`/fatwa/old-id` will return a 404). Only do this before the site is live.
- **Changing `date_added`**: This controls sort order on the homepage. Changing it moves the card in the listing.

### Does hot-reload work?

Yes, fully. Astro's dev server watches `src/content/fatwas/` and rebuilds the affected page(s) when you save. The browser refreshes automatically.

### Is there any manual step before deploying?

Yes — the same one as adding a fatwa: **run the search index script** if you want the search to reflect your edits.

```bash
npm run build:index
```

Or simply rely on `npm run build` which runs it automatically. On Vercel, every deploy runs `npm run build`, so search is always current after a deploy.

## 6. Dynamic vs. Hardcoded Pages & Components

Understanding what's "auto-generated" versus "manually defined" is key to knowing where to make changes.

### Fully dynamic — driven by content files

| Page / Feature | How it's generated |
|---|---|
| `/fatwa/[id]` | One page per `.md` file in `fatwas/`. Add a file, get a page. |
| `/category/[category]` | One page per **unique category string** found across all fatwa frontmatters. No manual registration needed. |
| Category listing (`/categories`) | Built at build time by scanning all fatwas and counting categories. |
| Homepage fatwa grid | All fatwas, sorted by `date_added`, rendered as cards. |
| Fatwa count badges on scholar cards | Counted dynamically from the collection at build time. |
| Search index (`search-index.json`) | Generated by `scripts/build-search-index.js` from all `.md` files. |

**Bottom line:** categories are 100% automatic. You never register a category anywhere — just use it in a fatwa's frontmatter and the system handles the rest.

### Partially dynamic — driven by `src/data/scholars.ts`

| Page / Feature | What's dynamic | What's hardcoded |
|---|---|---|
| `/scholars` | Fatwa counts per scholar | Scholar list, names, bios, Arabic names |
| `/scholar/[scholar]` | Fatwa grid content | One page generated per scholar in `scholars.ts` |
| `FilterBar` scholar pills | Renders all scholars from `scholars.ts` | The list of scholars itself |
| `[id].astro` breadcrumb | Scholar name looked up from `scholars.ts` | — |

**Bottom line:** scholar pages are _partially_ hardcoded. The slugs, names, and bios are defined in `src/data/scholars.ts`, not derived from content. To add a scholar you must edit that file AND the schema in `config.ts`. See [Section 12](#12-common-maintenance-scenarios).

### Fully hardcoded — only changes by editing source files

| Element | Where it lives |
|---|---|
| Site name "Fatāwā Archive" | `src/layouts/Layout.astro` (line with `<title>`) |
| Navigation links (Home, Scholars, etc.) | `src/layouts/Layout.astro` (the `<nav>` block) |
| Footer text | `src/layouts/Layout.astro` (the `<footer>` block) |
| Hero strip text on homepage | `src/pages/index.astro` |
| Site description in `<meta>` | Default is `'A collection of Islamic fatwas from senior scholars.'` in `Layout.astro` |
| Canonical URL domain fallback | `Astro.site ?? 'https://fatwas.example.com'` in `Layout.astro` — **you should update this** |

### The one hardcoded thing to fix before going live

In `src/layouts/Layout.astro`:

```ts
const canonicalURL = new URL(Astro.url.pathname, Astro.site ?? 'https://fatwas.example.com');
```

The `'https://fatwas.example.com'` fallback is a placeholder. Set your real domain in `astro.config.mjs`:

```js
export default defineConfig({
  site: 'https://fatawalib.com',   // ← add this line
  integrations: [tailwind()],
  output: 'static',
  // ...
});
```

Once `site` is set in config, the `??` fallback is never used.

## 7. Features That Exist Right Now

### 7.1 Homepage fatwa listing

**File:** `src/pages/index.astro`

All fatwas are fetched from the collection, sorted newest-first by `date_added`, and rendered as a 3-column responsive grid of `FatwaCard` components. Each card shows: category tags, the fatwa ID, the scholar name, the translator, and the date added.

The grid is rendered server-side (HTML is pre-built at deploy time). Filtering is done client-side via a small `<script>` block at the bottom of the page (no framework — plain JavaScript).

### 7.2 Scholar + category filters on homepage

**File:** `src/components/FilterBar.astro`

The filter bar has two parts:
- **Scholar pills**: one button per scholar, always shown. Clicking one sets `?scholar=ibn-baz` in the URL, which the client-side script reads and hides non-matching cards.
- **Category dropdown**: a `<select>` element populated from the complete list of categories (passed in as a prop from `index.astro`). Changing it sets `?category=prayer` in the URL.

These filters work by adding `data-scholar` and `data-categories` attributes to each card's wrapper `<div>` at build time. The JavaScript simply reads those attributes and shows/hides the divs. No server request is made when you filter.

**Limitation:** Scholar filter and category filter cannot be combined (selecting both simultaneously doesn't do AND-filtering in the current code — only one of them is used at a time based on which URL param is present).

### 7.3 Individual fatwa page

**File:** `src/pages/fatwa/[id].astro`

Each fatwa gets a full page at `/fatwa/{id}`. The page shows:
- Breadcrumb navigation (Home → Scholar → Fatwa #ID)
- A teal header card with: fatwa ID, scholar name with Arabic, all category tags, translator, optional reviewer, date issued, date added
- The rendered Markdown body (Question + Answer sections)
- A "View Original Source" button linking to `source_url`
- A "← Back to all fatwas" link

The Markdown body is rendered via Astro's built-in `render()` function, which converts Markdown to HTML. The `<Content />` component is placed inside a `<div>` with `prose` Tailwind Typography classes.

The page title tag is set to `Fatwa #001 — Shaykh Ibn Bāz` and the `<meta description>` is auto-generated as `A translated fatwa by Shaykh Ibn Bāz, covering: Prayer, Purification.`

### 7.4 Scholar listing page

**File:** `src/pages/scholars.astro`

Shows all 5 scholars as cards in a 2-column grid. Each card shows: Arabic name, transliterated name, lifespan, bio excerpt, and a badge showing how many fatwas that scholar has in the archive. Clicking a card goes to the scholar's individual page.

The fatwa count is live — it's recalculated at build time from the actual content files.

### 7.5 Individual scholar page

**File:** `src/pages/scholar/[scholar].astro`

One page per scholar (5 pages total: `/scholar/ibn-baz`, etc.). Shows the scholar's bio in a large header block, then a grid of all their fatwas sorted newest-first. If a scholar has zero fatwas, a "No fatwas yet" message is shown.

### 7.6 Category listing page

**File:** `src/pages/categories.astro`

A tag-cloud-style page showing every category that appears in at least one fatwa, sorted by frequency (most-used first). Each tag shows the category name and a count badge.

This page is 100% automatic — it builds itself from whatever is in the content files. You never touch this page when adding categories.

### 7.7 Individual category page

**File:** `src/pages/category/[category].astro`

One page per unique category. The URL slug is the category name lowercased with spaces replaced by hyphens: `"Hadith Sciences"` → `/category/hadith-sciences`. Shows a header with the category name and fatwa count, then a grid of fatwas.

### 7.8 Fuse.js search

**File:** `src/components/SearchWidget.astro` (loaded by `src/pages/search.astro`)

A dedicated search page at `/search`. See [Section 9](#9-search-implementation) for full details.

### 7.9 SEO basics

**File:** `src/layouts/Layout.astro`

Every page gets:
- `<title>` tag: `{page title} — Fatāwā Archive`
- `<meta name="description">` with either a passed-in description or the default
- `<link rel="canonical">` pointing to the page's canonical URL

What's **not** present: Open Graph tags (`og:title`, `og:image`), Twitter card meta, `sitemap.xml`, `robots.txt`, JSON-LD structured data.

### 7.10 Responsive layout

The entire site is mobile-responsive. The card grid collapses from 3 columns → 2 columns → 1 column. The navigation wraps on narrow screens. The filter bar stacks vertically on mobile.

### 7.11 Source attribution

Every individual fatwa page has a "View Original Source" button that links to the `source_url` in the frontmatter, opening in a new tab with `rel="noopener noreferrer"`.

## 8. Features That Are NOT Yet Implemented

These are the most obvious missing pieces — none require structural changes, just new files/components:

### 8.1 Pagination
The homepage loads all fatwas at once. With 5 fatwas this is fine, but with 200+ fatwas the page will become slow and long. You'd need to implement either static pagination (page 1, page 2...) or infinite scroll.

### 8.2 "View all X fatwas" count on homepage
There's no total count shown anywhere on the homepage. Users don't know how many fatwas are in the archive.

### 8.3 Related fatwas
The individual fatwa page has no "You might also like" or "Other fatwas in this category" section. Easy to add by filtering the collection for fatwas with overlapping categories.

### 8.4 RSS / Atom feed
Many readers subscribe to Islamic content sites via RSS. Astro has a built-in `@astrojs/rss` package that makes this easy to add.

### 8.5 `sitemap.xml`
No sitemap is generated. Astro has an official `@astrojs/sitemap` integration. Without a sitemap, search engines may take longer to index all fatwa pages.

### 8.6 `robots.txt`
No `robots.txt` file exists. Add one to `public/robots.txt` (Astro copies everything in `public/` to the build as-is).

### 8.7 Open Graph / social sharing meta tags
When someone shares a fatwa link on WhatsApp or Twitter, no preview card appears. You'd add `og:title`, `og:description`, and `og:url` tags inside `Layout.astro`'s `<head>`.

### 8.8 Dark mode
No dark mode toggle exists. Would require adding a `dark:` variant strategy in Tailwind and a toggle button in the nav.

### 8.9 Combined scholar + category filtering
As noted above, the current filter implementation can only apply one filter at a time. Combining "show only Ibn Bāz fatwas in the Prayer category" requires a small JavaScript update.

### 8.10 Fatwa titles / summary excerpts
Every fatwa card currently shows `Fatwa #001` as its title — not a descriptive title. The question text could be shown as the title, or you could add a `title` field to the schema. Right now the card doesn't show any preview of the question or answer text.

### 8.11 Arabic text display
There's no right-to-left (`dir="rtl"`) section for displaying the original Arabic text of fatwas. The `font-arabic` Tailwind class and Amiri font are loaded, so the infrastructure is there.

### 8.12 Admin / content management UI
All content management is done by editing files directly. There's no CMS, no login, no approval workflow. For a team situation, you'd look at Netlify CMS, Decap CMS, or Keystatic (which works with Astro Content Collections).

### 8.13 Comment or feedback section
No way for readers to flag translation errors or ask questions.

### 8.14 Print stylesheet
No print-specific styles. Adding `@media print` rules in `global.css` would let users print fatwas cleanly.

## 9. Search Implementation

### What library?

**Fuse.js** (version 7) — a lightweight JavaScript fuzzy-search library that runs entirely in the browser. No server required. The npm package is in `dependencies` in `package.json`.

### The two-part architecture

Search is split into two separate concerns:

**Part 1: The index (built at compile time)**

File: `scripts/build-search-index.js`

This Node.js script:
1. Reads every `.md` file in `src/content/fatwas/`
2. Parses each file with the `gray-matter` library to extract frontmatter and body
3. Strips Markdown syntax from the body (removes `##`, `**`, `*`, backticks, links) to get plain text
4. Builds a JSON array where each item looks like:
   ```json
   {
     "id": "001",
     "scholar": "ibn-baz",
     "scholarName": "Shaykh Ibn Bāz",
     "categories": ["Prayer", "Purification"],
     "translator": "Abu Abdillah",
     "reviewer": "Abu Khadeejah",
     "date_added": "2024-01-15T00:00:00.000Z",
     "body": "Question Is it permissible to pray while wearing shoes... Answer It is permissible..."
   }
   ```
5. Writes this array to `public/search-index.json`

This file is then served as a static asset — just a plain JSON file anyone can download.

**Part 2: The search widget (runs in the browser)**

File: `src/components/SearchWidget.astro`

When the `/search` page loads:
1. The browser fetches `/search-index.json` (one HTTP request)
2. Fuse.js is initialized with the downloaded data and configuration
3. A status line shows "X fatwas indexed" so the user knows it loaded
4. As the user types, Fuse.js searches the in-memory index and renders result cards as HTML strings

There is no server involved in searching. Every search query runs locally in the user's browser.

### When does the script run?

| Command | What happens |
|---|---|
| `npm run build` | Script runs automatically (first), then Astro builds the site |
| `npm run build:index` | Script runs alone — useful in development |
| `npm run dev` | Script does **not** run — uses whatever `search-index.json` already exists |
| Vercel deploy | `npm run build` is called, so script always runs fresh |

### Search configuration (how to tune it)

The Fuse.js configuration is in `SearchWidget.astro` inside the `<script>` block:

```ts
fuse = new Fuse(data, {
  keys: [
    { name: 'id',          weight: 0.1 },
    { name: 'scholar',     weight: 0.2 },
    { name: 'scholarName', weight: 0.2 },
    { name: 'categories',  weight: 0.3 },
    { name: 'body',        weight: 1.0 },  // ← body text is most important
    { name: 'translator',  weight: 0.1 },
  ],
  threshold:          0.35,   // 0.0 = exact match only, 1.0 = match anything
  includeMatches:     true,
  minMatchCharLength: 3,      // don't search until user types 3+ characters
  ignoreLocation:     true,   // don't penalize matches that appear late in the text
});
```

**To make search stricter** (fewer false positives): lower `threshold` toward `0.0`, e.g. `0.2`.

**To make search more forgiving** (catch more typos): raise `threshold` toward `0.5`, e.g. `0.4`.

**To weight categories more heavily** than body text: increase the `categories` weight above `1.0`.

**To show more results**: change `fuse.search(query, { limit: 20 })` — raise the limit.

**To require more characters before searching**: change `minMatchCharLength: 3` to a higher number.

The search input has a 220ms debounce (`setTimeout(..., 220)`) — this means the search only runs after the user stops typing for 220ms, avoiding unnecessary computation on every keystroke.

### Limitation to know about

The `body` field in the index contains the plain text of the fatwa **at the time the index was last generated**. If you edit a fatwa's body text and don't regenerate the index (`npm run build:index`), search results will show outdated text snippets (though clicking through to the fatwa page will show the current text).

## 10. Styling & UI

All custom colors are defined under `theme.extend.colors`. They're used via Tailwind utility classes like `bg-teal-900` or `text-gold-400`:

**To change the color scheme**, edit these hex values in `tailwind.config.js`. The names (`teal`, `gold`, etc.) are just labels — you could make `teal-900` actually green or navy without renaming it.

### Custom fonts (`tailwind.config.js`)

```js
fontFamily: {
  serif:  ['"Libre Baskerville"', 'Georgia', 'serif'],   // headings, fatwa titles
  sans:   ['"Source Sans 3"', 'system-ui', 'sans-serif'], // body text, UI labels
  arabic: ['"Amiri"', 'serif'],                           // Arabic text
},
```

These create Tailwind classes: `font-serif`, `font-sans`, `font-arabic`.

The actual fonts are loaded from Google Fonts in `Layout.astro`:
```html
<link href="https://fonts.googleapis.com/css2?family=Libre+Baskerville:ital,wght@0,400;0,700;1,400&family=Source+Sans+3:wght@300;400;500;600&family=Amiri:ital,wght@0,400;0,700;1,400&display=swap" rel="stylesheet" />
```

**To change fonts**: replace the Google Fonts URL and update the `fontFamily` values in `tailwind.config.js`.

### The `prose` / typography system

On the individual fatwa page (`fatwa/[id].astro`), the Markdown content is wrapped in:

```html
<div class="prose prose-fatwa prose-stone max-w-none
            prose-headings:font-serif prose-h2:text-teal-900 ...">
  <Content />
</div>
```

- `prose` — activates the `@tailwindcss/typography` plugin, which adds nice reading styles to plain HTML generated from Markdown
- `prose-fatwa` — applies the custom `fatwa` typography variant defined in `tailwind.config.js`
- `prose-stone` — sets the base prose color to stone (warm gray)
- `prose-headings:font-serif` — makes all headings use the serif font
- The remaining classes override specific elements inline

The custom `fatwa` typography variant is defined in `tailwind.config.js` under `theme.extend.typography`:

```js
fatwa: {
  css: {
    '--tw-prose-body':        theme('colors.stone[800]'),
    '--tw-prose-headings':    theme('colors.teal[900]'),
    '--tw-prose-links':       theme('colors.teal[800]'),
    '--tw-prose-quote-borders': theme('colors.gold[400]'),
    'h2': {
      fontFamily: theme('fontFamily.serif').join(', '),
      letterSpacing: '-0.01em',
    },
  },
},
```

### Global CSS (`src/styles/global.css`)

This file has three things:
1. The three Tailwind directives (`@tailwind base/components/utilities`) — required for Tailwind to work
2. A text selection color override (dark teal background, cream text)
3. A CSS rule to remove the top margin from the first `h2` inside a `.prose` block
4. A `.fatwa-card` component class (currently unused in HTML — you could apply it to cards instead of inline classes)

> **Note on the CSS import:** The global.css is imported in `Layout.astro` via `<style>@import url('/src/styles/global.css');</style>`. This works during development but is slightly non-standard. The more canonical way is to import it in the Astro frontmatter with `import '../styles/global.css'` — but the current approach works fine.

### How components are styled

Every component uses **only Tailwind utility classes** directly in the HTML — no separate `.css` files per component. This is standard Astro/Tailwind practice. All styling lives in the template itself, making components self-contained.

## 11. Deployment Notes

### `vercel.json`

```json
{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "installCommand": "npm install",
  "framework": "astro"
}
```

This tells Vercel:
- How to install dependencies (`npm install`)
- How to build the site (`npm run build`, which runs the search index script first, then Astro)
- Where the built files are (`dist/` — Astro always outputs here)
- What framework it is (`astro` — lets Vercel apply Astro-specific optimizations)

### Custom domain (fatawalib.com)

1. In the Vercel dashboard, go to your project → **Settings** → **Domains**
2. Add `fatawalib.com` and `www.fatawalib.com`
3. Vercel will show you DNS records to add at your domain registrar (usually an `A` record and a `CNAME`)
4. TLS/HTTPS is handled automatically by Vercel

After adding the domain, update `astro.config.mjs`:

```js
export default defineConfig({
  site: 'https://fatawalib.com',   // ← add this
  integrations: [tailwind()],
  output: 'static',
  // ...
});
```

This ensures canonical URLs in `<link rel="canonical">` use your real domain.

### Environment variables

**There are none currently required.** The site is entirely static — no API keys, no database connections, no secrets. The Anthropic API is not used in the deployed site.

If you add features later that need secrets (e.g. a contact form using an email API), you'd add environment variables in Vercel's dashboard under **Settings** → **Environment Variables**, and access them in Astro via `import.meta.env.YOUR_VAR_NAME`.

### Build output

`npm run build` produces a `dist/` folder containing only static HTML, CSS, and JS files. There is no Node.js server. Vercel serves these files from a CDN — pages load fast globally without any backend.

### Deployment workflow

The typical workflow:
1. Add/edit `.md` files in `src/content/fatwas/`
2. Push to your GitHub repository
3. Vercel automatically triggers a new build and deploy (if you've connected your repo)
4. The build runs `npm run build` which regenerates the search index and rebuilds all pages
5. In about 30–60 seconds, the new content is live at fatawalib.com

## 12. Common Maintenance Scenarios

### Add a new scholar

Requires changes to **two files**:

**Step 1 — `src/data/scholars.ts`**: add a new entry to the `scholars` object:

```ts
'new-scholar': {
  id:         'new-scholar',
  name:       'Shaykh New Scholar',
  arabicName: 'العالم الجديد',
  lifespan:   '1940–2010',
  bio:        'Brief biography here.',
},
```

Also add the new ID to the `ScholarId` type at the top:

```ts
export type ScholarId = 'ibn-baz' | 'ibn-uthaymin' | 'al-albani' | 'muqbil' | 'yahya' | 'new-scholar';
```

**Step 2 — `src/content/config.ts`**: add the slug to the enum:

```ts
const scholars = ['ibn-baz', 'ibn-uthaymin', 'al-albani', 'muqbil', 'yahya', 'new-scholar'] as const;
```

**Step 3 — `src/pages/scholars.astro`**: add the new scholar to the `counts` object:

```ts
const counts: Record<ScholarId, number> = {
  'ibn-baz':      0,
  'ibn-uthaymin': 0,
  'al-albani':    0,
  'muqbil':       0,
  'yahya':        0,
  'new-scholar':  0,  // ← add this
};
```

After these three changes, the new scholar appears on the `/scholars` page, gets their own page at `/scholar/new-scholar`, and their ID is valid in fatwa frontmatter.

categories:

### Change the sort order of fatwas

**File:** `src/pages/index.astro`, line:

```ts
const allFatwas = (await getCollection('fatwas')).sort(
  (a, b) => b.data.date_added.getTime() - a.data.date_added.getTime()
);
```

- Current: newest first (`b - a`)
- To sort oldest first: change to `a.data.date_added.getTime() - b.data.date_added.getTime()`
- To sort by scholar name: `a.data.scholar.localeCompare(b.data.scholar)`
- To sort by ID: `a.data.id.localeCompare(b.data.id)`

The same sort logic appears in `scholar/[scholar].astro` and `category/[category].astro` — update those too if you want consistent ordering everywhere.

Then in each fatwa's frontmatter:

```yaml

```

Then in `src/components/FatwaCard.astro`, replace:

```astro

```

with:

```astro

```
